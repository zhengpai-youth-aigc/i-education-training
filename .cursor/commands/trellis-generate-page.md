# 自适应生成或更新页面

根据用户提供的内容，先识别当前项目实际支持的页面类型与页面结构，再判断应该生成新页面、更新现有页面，还是把内容补充到已有页面中的合适位置。

## 适用输入

用户输入可能包括：
- 一段主题说明、课程大纲、资源清单
- 本地文件、文件夹、截图、文章草稿
- 网页链接、公众号/B站主页、站外文章
- 明确的修改需求（如“把这段内容加到学习渠道”）

## 核心原则

1. **先发现当前规范，再执行修改**。
2. **以用户明确要求优先**；用户未明确时，基于当前仓库真实结构给出推荐。
3. **不要把页面类型硬编码为固定几类**；每次执行时都要重新识别当前项目实际有哪些页面类型。
4. **优先使用本地已配置的 MCP / Skill**。
5. **网页检索优先用 `grok-search`**，不要优先用内置 WebSearch/WebFetch。
6. **代码与结构检索优先用 ace-tool、LSP、Read、Glob**。
7. **页面标题、主标题、摘要描述必须根据最终页面内容本身生成**；不要直接复用用户调用命令时的一句话任务描述。若页面主题是某个工具/概念，标题和摘要应先解释“它是什么、有什么作用、这页会解决什么问题”。
8. **如果输入材料中包含足够吸引人的价值点、结果感、对比数据或收益指标，必须在首屏或前两屏优先展示**，例如：配置后会有什么变化、能节省多少时间/成本/token、能带来什么结果。不要把这些高吸引力信息埋到后文。
9. **不要把生成过程、抓取过程、检索失败、来源缺失、仓库内重组、命令执行策略等内部说明写进最终页面正文**。这类信息只允许出现在你给用户的结果汇报里，不允许暴露给最终页面读者。
10. **资源页、工具页、推荐集合页如果不属于正式课程，不得占用课程编号语义**。即使历史文件名带有 `lesson-*`，也要优先按当前信息架构修正为非课程处理方式，并避免影响后续课程排序、进度和编号扩展。

## 步骤

### 1. 收集输入上下文

先识别用户输入的来源与形态：文本、路径、链接、图片，还是混合输入。

优先处理方式：
- 本地文件/目录：用 `Read`、`Glob`、LSP、ace-tool 读取和理解
- 项目结构/现有页面模式：优先用 ace-tool、LSP、`Read`
- 外部网页/链接：优先用 `grok-search` 的 `web_fetch` / `web_search`

### 1.1 图片输入分类（新增必做）

当输入中包含图片（截图、海报、长图、PPT页、聊天记录截图、界面截图）时，不要把所有图片一律当成“素材图”处理，必须先判断图片在本次页面中的主要作用。

至少分为两类：

- **解释类图片**：图片里承载了大量文字说明、步骤、结论、推荐理由、表格信息，页面读者真正需要的是其中的内容本身，而不是图片这个外观。
- **展示类图片**：图片主要用于展示界面、效果、案例、排版风格、产品形态、前后对比，页面读者需要“看图感受”，图片本身应保留。

可用以下启发式判断：
- 若图片中文字很多、结构接近文档/PPT/长截图/说明海报，优先判为**解释类图片**
- 若图片重点是 UI、产品界面、视觉效果、案例展示，优先判为**展示类图片**
- 若图片两者兼有，按“用户最终阅读页面时，真正需要复制的是文字信息还是视觉感受”来定主类型

### 1.2 图片处理策略（新增必做）

对不同类型的图片，处理策略必须不同：

**解释类图片**
- 优先提取其中的关键信息、标题层级、步骤、列表、表格结论
- 将提取结果与同目录下的文稿、链接、说明文字一起整理成页面正文
- 默认**不要**把大量解释类图片原样整批塞进页面，否则会让页面变成“截图堆叠”而不是可读内容
- 如需保留原图，也只保留 1-2 张最有代表性的作为辅助说明，并补充明确 `alt` 文本

**展示类图片**
- 保留图片作为视觉素材，在页面中按现有样式放入合适的位置
- 优先用于：案例展示区、步骤示意区、前后对比区、工具界面示例区
- 若同类展示图过多，必须先筛选，避免整页连续堆图；优先保留最能支撑正文观点的少量图片
- 图片应服务于正文结构，而不是让正文围着图片转

如果本次输入是“图文混合文件夹”，默认策略应为：
- **解释类图片 → 提炼成文字内容**
- **展示类图片 → 少量精选后参与排版**

若当前环境中的多模态图片理解结果不稳定、无法可靠读出图内关键信息，退回保守模式：
- 以文稿、文件名、相邻文本、链接说明为主组织页面
- 图片仅作辅助参考，不强行抽取图内细节
- 最终优先保证页面内容正确、可读，而不是勉强“把图全用上”

在真正修改前，必须先读取并遵循当前项目中的最新规范，而不是依赖旧记忆。

至少检查：
- `.trellis/spec/frontend/index.md`
- `.trellis/spec/frontend/component-guidelines.md`
- `.trellis/spec/frontend/hook-guidelines.md`
- `.trellis/spec/frontend/quality-guidelines.md`
- `.trellis/spec/guides/index.md`

然后检查当前真实页面结构，重点识别：
- `public/` 下有哪些页面文件
- `_site/` 下哪些页面是当前构建生成的部署输出
- `site/`、`public/` 里哪些文件分别是真实页面来源与共享资源来源
- 现有页面之间有哪些命名模式、编号模式、入口模式、导航模式

### 3. 动态识别当前项目的页面类型

不要预设“只有课程页、资源页、学习渠道推荐”这三类。

你需要根据当前仓库现状，总结出项目当前真实存在的页面类型，例如：
- 课程页
- 资源页
- 推荐集合页
- 专题页
- 工具页
- 落地页
- 文章页

识别方式应基于：
- 页面文件名模式
- 首页入口分组
- 侧边栏分组
- 页面内部结构（章节式、卡片式、推荐列表式、文章式等）

### 4. 将用户输入映射到当前页面类型

根据输入内容与当前发现的页面类型，给出最合适的映射：

- 如果用户已明确指定页面类型，按用户要求执行
- 如果用户未明确指定，则先给出 **推荐类型 + 理由**
- 如果内容无法稳定归类，则给出 2-3 个最合适的落点供用户选择

推荐时优先考虑：
- 内容是系统化教学，还是资料整理，还是推荐条目补充
- 当前项目里是否已有同类页面结构
- 修改后是否需要被首页、侧边栏、搜索、上下页导航纳入

### 5. 检查命令定义是否过期

如果你发现当前项目结构已经明显超出本命令原有假设，例如：
- 出现了新的页面大类
- 导航/搜索同步规则发生了结构性变化
- 页面镜像关系从 `public`/`docs` 变成了别的模式

那么：
- **先按当前代码真实结构完成本次任务**
- 然后在结果中提示：`/trellis:generate-page` 的命令定义可能需要更新
- 默认**不要自动重写命令文件本身**
- 只有当用户明确要求“更新这个命令自身”时，才去修改：
  - `.claude/commands/trellis/generate-page.md`
  - `.cursor/commands/trellis-generate-page.md`

### 6. 按现有模式修改页面

修改时必须以当前仓库已有模式为准：

- 若是新增页面，优先复用最接近的现有页面结构
- 若是更新已有页面，优先沿用原有 HTML 结构、CSS class、脚本模式
- 若新增页面需要被导航到，补齐所有必要入口
- 若页面会被站内搜索命中，补齐搜索数据
- 若页面带有上一页/下一页或进度逻辑，按当前规则同步处理
- 若页面属于发布内容，更新真实来源并验证重新构建后的 `_site/` 输出

特别检查这些常见同步点：
- 侧边栏导航
- 首页入口区块
- 首页“最新动态”（Latest updates）区块；如果新增/更新的页面值得作为最新入口展示，必须同步更新首页该区块的排序或条目文案
- 搜索弹窗中的页面数据
- 上一页/下一页导航
- 课程完成按钮或进度逻辑
- 构建后的 `_site/` 输出与真实来源的对应关系

### 6.1 图文混合素材的页面组织规则（新增必做）

当输入同时包含 markdown / 文稿 / 链接 / 图片时，页面组织优先级如下：

1. 先用文本材料和解释类图片提炼出页面的核心结构（标题、摘要、小节、结论）
2. 再从展示类图片里挑选少量最有价值的图片插入相应小节
3. 若图片信息无法稳定提炼，宁可少放，也不要为了“素材都用上”而牺牲页面可读性
4. 最终页面必须在**不看大部分原图**的前提下依然可完整阅读

特别注意：
- 如果用户给的是课程草稿或分享材料，最终页面应优先做成“可讲、可读、可搜索”的课程内容，而不是简单的素材归档页
- 如果解释类图片里的信息和文本稿重复，正文以更顺畅、更适合阅读的整理稿为准，不必机械重复
- 若你选择不展示某些图片，可以在最终汇报里说明它们被当作解释类素材吸收进正文，但不要把这类内部处理说明写进页面正文

### 6.2 成文口吻约束（新增必做）

最终页面正文必须是**面向读者的成品表达**，不能暴露你的素材来源或生成过程。

禁止在最终页面正文里出现这类表述：
- “这批草稿提到”
- “用户给的素材里有”
- “根据输入内容整理”
- “本次任务里”
- “从图片中提取到”
- 任何类似“我/你/这次生成/本页整理过程”的内部工作流表述

允许的做法是：
- 直接把内容写成结论、建议、解释、步骤、推荐
- 把来源材料消化后再表述，像正常作者写文章一样落笔
- 如果确实需要交代背景，也要写成读者视角，例如“常见做法是……”“更适合新手的顺序是……”

判断标准：
- 页面读者不应该意识到这是一篇“由素材拼出来的页面”
- 页面应该读起来像一篇原生课程稿、专题稿或资源页，而不是生成记录

### 7. 仅在有歧义时请求确认

如果归类非常明确，直接说明判断并执行。

如果存在歧义，用下面格式回复：

- 推荐放到：`<动态识别出的页面类型>`
- 原因：<为什么按当前项目规范更适合这里>
- 可选方案：<1-2 个备选>
- 接下来会修改的文件：<文件列表>

然后请用户确认。

### 8. 命令自检模式（每次必做）

在本次页面处理完成后，必须额外做一次命令层面的自检，判断这个命令本身是否仍然贴合项目现状。

至少检查：
- 本次是否识别到了此前命令未覆盖的新页面类型
- 本次是否发现了新的入口分组或导航分组
- 本次是否发现了新的同步点（如搜索、镜像、进度、上下页）
- 本次是否出现了新的页面命名模式、编号模式、文件组织模式

自检结论规则：
- 若没有发现结构性变化，结论写：`命令定义仍适用`
- 若发现轻微变化但本次仍可兼容，结论写：`命令定义基本适用，建议后续更新`
- 若发现明显超出命令原假设的变化，结论写：`命令定义可能过期，建议更新命令`

### 8.1 页面内容与排版自检（每次必做）

在汇报前，必须额外检查一次页面内容是否真正贴合输入，以及页面是否出现明显排版问题。

至少检查：
- 页面 `<title>`、首屏 `h1`、首段摘要，是否是基于页面实际内容提炼，而不是照搬任务描述
- 如果主题是工具、概念、项目、教程，标题和摘要是否先说明“它是什么 / 有什么作用”
- 如果素材中存在高吸引力信息（效果、收益、节省比例、前后对比），是否已经前置到首屏或前两屏
- 在桌面宽度下是否出现内容卡片超出页面主体、横向溢出、代码块把布局撑坏
- 在窄屏下网格是否能自然折叠，是否仍可阅读

若发现标题摘要失真或页面溢出，必须先修正，再进入最终结果汇报
